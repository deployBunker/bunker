Project Bunker

Multi-agent compute isolation platform — spin up per-agent Linux environments with their own rootless Docker daemon, resource limits, and networking. One API call.

🔗 github.com/deployBunker/bunker 📜 Apache 2.0 🟢 Go 1.25+ 📡 Deployed on Hetzner

Executive Summary

Bunker is a multi-agent hosting platform that gives every AI agent its own isolated Linux environment — complete with a private rootless Docker daemon, CPU/memory/disk/process limits enforced via cgroups, SSH access, and optional public networking via Cloudflare tunnels or Tailscale.

Instead of one VM per agent (expensive, slow to provision, 80% idle), Bunker dense-packs 50+ agent hosts onto a single bare-metal server. Each agent lives minutes to hours — spun up on demand, auto-destroyed on TTL expiry. Clean teardown: userdel -r and everything is gone.

Problem & Solution

🔴 The Problem

  • AI coding agents need Docker to build, test, and run code
  • Sharing a single Docker daemon across agents is a security and isolation nightmare
  • One VM per agent costs $20–80/month each and leaves idle VMs accumulating
  • Docker-in-Docker requires --privileged and breaks isolation
  • Manual provisioning of agent environments is slow and error-prone
  • No standard way to limit agent resource usage (CPU, memory, disk)

🟢 Bunker's Solution

  • Per-agent Linux user with its own rootless Docker daemon
  • cgroups v2 enforce CPU, memory, disk, process, and container limits
  • One $80/mo bare-metal server hosts 50+ concurrent agents
  • Single CLI call: bunker spawn --cpu 2 --mem 4g --ttl 6h
  • Auto-destroy on TTL expiry — no forgotten VMs
  • gRPC + REST API with JWT auth, TLS/mTLS, per-agent scoped keys
  • Native SSHFS mount, Docker tunnel, Cloudflare tunnels, Tailscale mesh

Key Features

🐳 Agent Isolation

Each agent is a dedicated Linux user with its own rootless Docker daemon. No Docker-in-Docker. No --privileged. Complete isolation via user namespaces + cgroups v2.

📊 Resource Limits

CPU quota, memory max, disk quota, process count, open file limit, and Docker container limit — all enforced at the cgroup level via systemd user slices.

🔗 SSH Transport

Primary Docker access via native DOCKER_HOST=ssh://agent@host. SSHFS for filesystem mounts. SSH tunnel fallback for tools requiring TCP Docker sockets.

🌐 Networking Stack

Three modes: Cloudflare Tunnels (named or TryCloudflare), Tailscale mesh, or direct port ranges. Per-agent port ranges prevent collisions.

⏱️ TTL & Heartbeat

Agents auto-destroy on TTL expiry. Heartbeat extends TTL. No forgotten agents burning resources. Clean userdel -r teardown.

🔐 Security

JWT auth (HS256/RS256/Ed25519), agent-scoped opaque sub-keys, TLS via certmagic (self-signed or Let's Encrypt), mTLS support, master-only RPC enforcement.

🖥️ CLI Surface

Full cobra CLI: connect, spawn, list, info, exec, run (--detach), metrics, mount, tunnel, heartbeat, destroy, env (set/get/list/unset). Multi-server support.

🔧 Developer UX

bunker exec --raw for no-shell interpretation. bunker exec --script for file upload. bunker run --detach for persistent background processes via systemd transient units.

🔑 API Key Management

Master tokens for admin. Agent-scoped sub-keys for CI/CD pipelines. Keys can only access their own agent's scoped RPCs.

Resource Limit Enforcement

All limits enforced at two levels: systemd user slice (cgroups v2) and Docker unit properties.

  rpc ServerInfo(ServerInfoRequest) returns (ServerInfoResponse);
  rpc ServerMetrics(ServerMetricsRequest) returns (ServerMetricsResponse);
  rpc SpawnAgent(SpawnAgentRequest) returns (SpawnAgentResponse);
  rpc DestroyAgent(DestroyAgentRequest) returns (DestroyAgentResponse);

  rpc RunAgent(RunAgentRequest) returns (RunAgentResponse);

// Agent service — scoped sub-key access only

  rpc GetInfo(GetInfoRequest) returns (GetInfoResponse);
  rpc Metrics(AgentMetricsRequest) returns (AgentMetricsResponse);
  rpc Heartbeat(HeartbeatAgentRequest) returns (HeartbeatAgentResponse);
}

Use Cases

🤖 AI Coding Agents

coding-hermes and similar autonomous coding agents need Docker to build, test, and run code. Bunker gives each agent its own clean Docker host — bunker spawn --cpu 4 --mem 8g --ttl 6h → DOCKER_HOST=ssh://agent-7@srv1 docker build . → destroy when done.

🔄 CI/CD Sandboxes

Spin up ephemeral build environments per PR or branch. Agent-scoped API keys mean CI pipelines can only access their own agent. TTL ensures cleanup even if the pipeline crashes.

📦 Multi-Tenant SaaS

Each tenant gets their own agent with resource limits and private Docker daemon. Cloudflare tunnels provide public URLs. Bunkerd manages the fleet.

🔬 Development Environments

On-demand dev environments with bunker spawn. SSHFS mount for local IDE. Docker tunnel for container tooling. env set for configuration injection.

🏠 Homelab Agent Hosting

Run on a home server behind NAT. Cloudflare tunnels handle public access. Tailscale for private mesh networking. One daemon manages all agents.

⚡ Serverless Containers

Bunker as a lightweight alternative to Fargate/Cloud Run. Per-agent Docker daemon means no cold starts from image pulls. Persistent home directories between runs.
